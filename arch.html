Jarvis Assistant — Architecture & System Design

Hybrid local-first AI assistant with orchestration, multi-model routing, and safety.

High-level model architecture

Model portfolio & routing

The Jarvis assistant uses a portfolio of models, each with a specific role. The system first understands the user's intent and task complexity using a lightweight local "router" model and heuristics. Then it selects the most appropriate execution path and model.

Local router model

A small local model (or the general model in a special "classifier mode") reads the prompt and classifies: intent (general / coding / complex) and complexity (easy / medium / difficult). This meta decision controls routing rather than answering the question directly.

General models

A fast local general model handles everyday chat, Q&A, summaries, and planning. For medium or difficult general tasks, a stronger local model can be used if the hardware supports it, still keeping data on-device.

Coding models

A local coding model focuses on code generation, debugging, refactoring, and repo-aware tasks. It is paired with coding tools (repo scanner, file reader/writer, test runner, shell command runner with guardrails) to form a coding agent.

Cloud models (complex path)

For difficult tasks that exceed local capacity (deep system design, multi-constraint optimization, long research), the system escalates to a cloud model chain via OpenRouter. A fallback mechanism routes back to local models if cloud calls fail.

Intent: general / coding / complex
Complexity: easy / medium / difficult
Routing: rules + router LLM
Fallback: cloud → local
System design

Layered system architecture

The assistant is designed as a layered system. Each layer has a clear responsibility and communicates with its neighbors via well-defined interfaces. This makes the system easier to reason about, test, and extend.

User | v Interface layer (Streamlit UI) - Chat UI, session history - Voice (future), file upload | v Gateway layer (FastAPI) - HTTP API (/chat, /health) - Input validation, logging, streaming - Input/output guardrails | v Orchestration layer (LangGraph) - Shared state (intent, complexity, path, model, context) - Router node (rules + router LLM) - Context builder (memory + retrieval) - Risk checker & approval gates - Fallback handler | v Execution layers ├─ General path (local general model + general tools) ├─ Coding path (local coding model + coding tools) └─ Complex path (OpenRouter cloud chain) | v Memory & Retrieval - Short-term chat history - Long-term user memory - Vector DB (documents, embeddings) | v Safety & Policy - Prompt injection checks - PII redaction - Risk classification & permissions
Routing & model selection

Intent & complexity detection

Intent detection is implemented as a hybrid process. Simple, obvious cases are classified using deterministic rules. Ambiguous cases are delegated to a small local router LLM that returns a JSON classification. This combines reliability with flexibility.

Heuristic rules

  • Keyword patterns: coding words (bug, error, repo) → coding intent.
  • Complex phrases (architecture, optimize, multi-objective) → complex intent.
  • Prompt length: short → easy; very long → difficult.

Router LLM

For borderline prompts, the router LLM is asked to classify intent and complexity explicitly. It responds with JSON like: 

{
  "intent": "complex",
  "complexity": "difficult"
}

The orchestration graph stores these labels in state and uses them to choose the branch and model portfolio for execution.

Model selection

The combination of intent and complexity determines which model to use:

  • General + easy → small local general model.
  • Coding + medium → local coding model + tools.
  • Complex + difficult → cloud path via OpenRouter.
Workflow

End-to-end request workflow

This is how a single user request flows through the system, from input to final response, including safety checks and possible escalation to cloud models.

1 User enters a message in the Streamlit UI. The UI sends the message and session ID to the FastAPI backend.
2 FastAPI validates the payload, applies input guardrails, and logs the request. If the input is blocked (e.g., prompt injection), it returns a 400 error.
3 The orchestration graph is invoked with a fresh state: user input, session ID, and any existing history.
4 The router node runs: heuristics first, then the router LLM (if needed), to assign intent and complexity.
5 The context builder fetches relevant memory and documents from the vector DB and stores them in retrieved_context.
6 The graph uses the intent and complexity to choose the execution branch and model (e.g., general-local, coding-local, or complex-cloud).
7 If tools are enabled for that branch, the model can call tools through the orchestration layer. Tool calls are logged and can be gated by risk classification.
8 High-risk actions (like file writes or terminal commands) trigger an approval gate, pausing the graph until the user approves in the UI.
9 The selected model generates a response, which is then passed through output guardrails (e.g., PII redaction) and returned to the frontend.
10 The frontend displays the response along with metadata (path used, model used), and the state can be updated for the next turn.
Summary

Design goals

The Jarvis assistant is designed to be local-first, with cloud escalation only when necessary. Its architecture emphasizes modularity (clear layers), safety (guardrails and approvals), and adaptability (multi-model routing based on intent and complexity). The orchestration layer, built with LangGraph, acts as the "brain" that coordinates all components.
