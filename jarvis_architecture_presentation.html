Architecture stack

1. Interface layer

Streamlit chat UI, file upload, mode indicators, session history, and later voice input and text-to-speech output. This is where the user sees routing, approvals, and tool activity.

2. Gateway layer

FastAPI receives requests, validates schemas, applies input guardrails, starts traces, and streams clean responses back to the frontend.

3. Orchestration layer

LangGraph manages state, routing, handoffs, retry logic, approvals, and the ordered execution of tools, memory, and model calls.

4. Execution layers

Model portfolio, memory and retrieval, tools, coding-agent subsystem, and safety policy all sit below orchestration as specialized execution domains.

Core idea

Why orchestration is mandatory

State control

The orchestrator stores session state, retrieved context, selected path, active tools, approval flags, and final response.

Workflow control

It decides whether the request should go through a deterministic workflow, a coding workflow, or a complex cloud branch.

Recovery control

It handles retries, provider failover, branch handoff, and fallback to local models when cloud routes are unavailable.

Best pattern

Use a deterministic outer graph with agentic behavior only inside controlled nodes. That keeps the system reliable while still allowing flexible tool use and coding workflows.

How we orchestrate

LangGraph node flow

  • 1. Validate input and session metadata.
  • 2. Classify intent: general, coding, or complex.
  • 3. Build context from memory and retrieval.
  • 4. Select model path and tool permissions.
  • 5. Execute branch-specific tools and model calls.
  • 6. Check risk, approval, output validation, and fallback.

Orchestration state schema

  • user_input, session_id, history
  • intent, selected_path, selected_model
  • retrieved_context, tool_calls, tool_results
  • risk_level, approval_required
  • fallback_count, error_state, final_response

Path routing

Three execution branches

General

For normal chat, planning, summaries, retrieval and personal assistant flows. Default local path for low cost and privacy.

Coding

For repo-aware engineering tasks such as reading files, patching code, searching symbols, running tests and controlled shell commands.

Complex

For long reasoning, advanced architecture decisions or hard coding problems that exceed local capability and must escalate to OpenRouter.

Routing should be rule-first with optional small-model classification for ambiguous requests. That keeps latency predictable and makes debugging much easier.

Safety

Protection and approval flow

Before execution

Validate input, detect prompt injection, classify task risk, and decide whether the task can be auto-run or must stop for approval.

After execution

Validate output, remove sensitive data, record traces, and send either a clean response or a structured warning with recovery options.

01

Check if the action is read-only, write, execute, or external.

02

Pause the graph if approval is required.

03

Resume the exact node path after approval without losing state.

🧠

Final architecture direction

Build a local-first Jarvis with LangGraph orchestration, specialized local execution branches, strong safety gates, and OpenRouter only for truly complex escalation. That gives you the best balance of reliability, privacy, cost control and explainability.
